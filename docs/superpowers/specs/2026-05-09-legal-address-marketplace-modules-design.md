# Дизайн модулей маркетплейса юридических адресов

Дата: 2026-05-09

## Цель

Развить существующий сервис юридических адресов в маркетплейс, где:

- клиенты выбирают юридический адрес, создают заявку, получают личный кабинет и документы;
- собственники помещений могут отправить запрос на подключение, а после приглашения администратора получают кабинет исполнителя;
- администратор площадки проверяет запросы, адреса, заявки, документы и вручную управляет спорными переходами;
- каждый этап разработки завершается отдельной git-версией.

Проект уже содержит FastAPI-бэкенд, React/Vite-фронт, авторизацию для команды, модели собственников, адресов, заявок, клиентов, файлов и генерации документов. Новый дизайн должен развивать существующую структуру, не переписывая рабочие части без необходимости.

## Утвержденные продуктовые решения

1. Разработка идет гибридно: каждый этап дает видимый продуктовый срез и одновременно укрепляет модель данных.
2. В MVP используются реальные роли и логин.
3. Клиентский аккаунт создается при отправке заявки: каталог открыт без входа, но заявка сразу привязывается к пользователю с ролью `client`.
4. Исполнитель попадает на площадку через гибридный onboarding: собственник может отправить публичный запрос на подключение, но доступ выдает администратор через приглашение.
5. Публикация адресов и выдача документов требуют ручной модерации.
6. В MVP платеж может быть ручным или имитированным статусом; полноценный escrow и split-платежи откладываются.

## Роли

### Client

Клиент сервиса: учредитель, директор, бухгалтер, юрист или регистрационный агент. Может смотреть каталог, создать заявку, войти в личный кабинет, отслеживать статус, отвечать на запросы правок и скачивать документы.

### Owner / Provider

Собственник помещения или уполномоченный представитель. Может запросить подключение, после приглашения заполнить профиль, управлять своими адресами, принимать заявки, загружать документы и видеть историю работ.

### Manager / Lawyer / Admin

Команда площадки. Существующие роли сохраняются, но админка получает новые операции:

- проверка запросов собственников;
- приглашение исполнителей;
- модерация адресов;
- проверка клиентских заявок;
- назначение заявки исполнителю;
- проверка документов;
- ручная смена статусов, отмены и споры.

## Модули MVP

### 1. Системное ядро маркетплейса

Добавляет основу, от которой зависят остальные модули:

- роли `client` и `owner` / `provider`;
- связь пользователя-исполнителя с `Provider`;
- связь пользователя-клиента с заявками;
- маркетплейс-статусы заявки;
- журнал событий уведомлений;
- типы загружаемых документов;
- seed-данные для демонстрационного потока.

Существующая логика генерации гарантийных писем и договоров остается, но новые статусы должны поддерживать маркетплейс-процесс до и после генерации документов.

### 2. Публичная зона и каталог адресов

Открыта без входа. Содержит:

- публичную главную / витрину;
- каталог доступных адресов;
- фильтры по городу, ИФНС, цене, сроку, почтовому обслуживанию и доступности;
- карточку адреса с условиями, ценой, собственником в обезличенном или кратком виде;
- кнопку создания заявки;
- форму запроса собственника на подключение.

Каталог использует существующие `addresses` и `providers`, но отображает только опубликованные и доступные адреса.

### 3. Форма заявки клиента

Путь клиента:

1. Выбор адреса из каталога.
2. Выбор типа заявки: первичная регистрация или смена адреса.
3. Заполнение данных компании / будущей компании и контактов.
4. Создание или вход в клиентский аккаунт в процессе отправки заявки.
5. Создание заявки в статусе `draft` или `awaiting_payment`.
6. Создание первого события уведомления для клиента и админа.

Для MVP допустимо ручное подтверждение оплаты: администратор переводит заявку в `paid`.

### 4. Личный кабинет клиента

Клиент видит только свои заявки:

- список заявок;
- карточку заявки с адресом, типом, ценой, текущим статусом и историей;
- комментарии администратора при статусе `needs_client_fix`;
- документы, доступные после `ready_for_client`;
- уведомления по изменениям статуса.

Клиентские действия в MVP:

- открыть заявку;
- внести правки по запросу администратора;
- скачать доступные документы;
- подтвердить получение или открыть спор.

### 5. Личный кабинет исполнителя

Исполнитель видит только связанные с ним адреса и заявки:

- профиль собственника / организации;
- адреса и их статус модерации;
- входящие заявки;
- принятие или отклонение заявки с причиной;
- перевод в подготовку документов;
- загрузка документов;
- просмотр комментариев администратора при доработке.

Публикация нового адреса исполнителем в MVP разрешена только через модерацию администратора.

### 6. Админка и ручная модерация

Админка управляет всеми контрольными точками:

- запросы собственников на подключение;
- приглашения исполнителей;
- модерация профиля исполнителя;
- модерация адреса;
- проверка заявки клиента;
- ручное подтверждение оплаты;
- назначение заявки исполнителю;
- проверка загруженных документов;
- возврат заявки клиенту или исполнителю на доработку;
- отмена, спор, завершение.

Админка должна показывать очереди: новые запросы, заявки на проверке, документы на проверке, спорные заявки.

### 7. Загрузка документов

Используется существующая инфраструктура `StoredFile`, расширенная типами документов и привязкой к заявке:

- документы клиента: реквизиты, данные компании, дополнительные файлы;
- документы исполнителя: подтверждение права на помещение, гарантийное письмо, договор, акт, согласие, почтовое обслуживание;
- служебные документы администратора: проверенная версия, комментарии, решение по спору.

В MVP достаточно загрузки файлов, списка файлов, скачивания и отметки проверки. Электронная подпись, автоматическое распознавание и автогенерация всех документов откладываются.

### 8. Базовая логика уведомлений

В MVP уведомления реализуются как внутренний журнал событий:

- событие создается при создании заявки;
- при смене статуса;
- при запросе правок;
- при загрузке документов;
- при одобрении документов;
- при споре или отмене.

Фронт показывает индикатор непрочитанных событий и список уведомлений в кабинетах. Реальная отправка email/Telegram/SMS откладывается, но структура событий должна позволять подключить внешние каналы позже.

## Статусы заявки

Основная цепочка:

1. `draft` — заявка создана или заполнена не полностью.
2. `awaiting_payment` — заявка ожидает оплаты или ручного подтверждения счета.
3. `paid` — оплата подтверждена.
4. `admin_review` — администратор проверяет заявку.
5. `assigned_to_owner` — заявка передана исполнителю.
6. `accepted_by_owner` — исполнитель принял заявку.
7. `documents_preparing` — исполнитель готовит документы.
8. `documents_uploaded` — исполнитель загрузил комплект.
9. `documents_review` — администратор проверяет документы.
10. `ready_for_client` — документы доступны клиенту.
11. `completed` — заявка завершена.

Боковые статусы:

- `needs_client_fix` — нужны правки клиента;
- `documents_revision` — документы возвращены исполнителю на доработку;
- `rejected_by_owner` — исполнитель отклонил заявку;
- `cancelled` — заявка отменена;
- `dispute` — спорная ситуация;
- `refund_pending` — требуется возврат;
- `refunded` — возврат выполнен.

Статусы должны храниться как единый источник правды на бэкенде. Фронт отображает разные доступные действия по роли пользователя и текущему статусу.

## Тестовые данные

Seed-набор нужен для демонстрации полного пути:

- администратор;
- менеджер или юрист;
- два исполнителя;
- несколько адресов в разных городах / ИФНС;
- один опубликованный адрес, один адрес на модерации, один недоступный адрес;
- клиент с одной активной заявкой;
- заявка на каждом ключевом этапе;
- набор файлов-заглушек для документов.

Seed-команда должна быть повторяемой и безопасной для локальной разработки.

## MVP

Входит в MVP:

- реальные роли и вход;
- публичный каталог адресов;
- запрос собственника на подключение;
- приглашение исполнителя администратором;
- клиентская заявка с созданием аккаунта;
- личный кабинет клиента;
- личный кабинет исполнителя;
- админка с ручной модерацией;
- маркетплейс-статусы заявки;
- загрузка и скачивание документов;
- внутренние уведомления;
- seed-данные;
- тесты для ключевых API и статусных переходов.

Откладывается:

- онлайн-эквайринг;
- escrow и split-платежи;
- электронная подпись;
- автоматическая проверка ГАР / ФНС;
- автоматическое распознавание документов;
- полноценный чат;
- публичные рейтинги исполнителей;
- API для регистрационных агентств;
- email/SMS/Telegram-рассылки;
- сложная аналитика и SLA-дашборды.

## Порядок git-этапов

Каждый этап должен завершаться проверкой и отдельным коммитом:

1. `feat: add marketplace core model`
   - роли, статусы, связи, события, seed-основа.
2. `feat: add public address catalog`
   - публичная витрина, каталог, фильтры, запрос собственника.
3. `feat: add client application flow`
   - форма заявки, создание клиентского аккаунта, стартовые уведомления.
4. `feat: add client dashboard`
   - кабинет клиента, заявки, история, документы.
5. `feat: add owner dashboard`
   - кабинет исполнителя, адреса, входящие заявки, загрузка документов.
6. `feat: add admin moderation workflow`
   - очереди проверок, ручные переходы, проверка документов.
7. `feat: add notification center and demo polish`
   - уведомления, тестовые сценарии, сборка, финальная проверка.

Если этап оказывается слишком большим, он дробится на подэтапы, но каждый подэтап также завершается коммитом.

## Технические границы

### Backend

- FastAPI-роутеры остаются основным API-слоем.
- Новые публичные endpoints должны быть явно добавлены в список public-path в middleware.
- Ролевые проверки должны жить на бэкенде, а не только во фронте.
- Миграции Alembic обязательны для изменений схемы.
- Новые статусные операции лучше оформлять командами API, а не свободным обновлением поля `status`.

### Frontend

- Текущий `frontend/src/App.tsx` слишком крупный. Новые модули нужно выносить в отдельные компоненты и, по возможности, в отдельные файлы.
- Навигация должна разделять публичную зону, кабинет клиента, кабинет исполнителя и админку.
- Экран по умолчанию для публичного пользователя — каталог, а не внутренняя CRM.
- Интерфейсы должны быть плотными и рабочими: каталог, таблицы, статусы, очереди, формы, файлы.

### Проверка

Минимум для каждого этапа:

- backend-тесты для новой бизнес-логики;
- frontend build;
- ручная проверка главного сценария в браузере;
- `git status` перед коммитом, чтобы не смешивать временные файлы companion.

## Открытые решения, принятые по умолчанию

- Роль исполнителя называется `owner` на уровне продукта; в коде можно связать ее с существующей сущностью `Provider`, чтобы не ломать текущие модели.
- Ручная оплата в MVP считается статусным переходом, а не интеграцией с платежным провайдером.
- Запрос собственника на подключение не создает активного пользователя автоматически. Пользователь исполнителя появляется после приглашения администратора.
- Клиентский пользователь создается при заявке и сразу получает доступ только к своим данным.
